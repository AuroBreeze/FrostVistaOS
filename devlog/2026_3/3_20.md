> Here is the original text:
> An ELF header resides at the beginning and holds a ``road map'' describing the file's organization. Sections hold the bulk of object file information for the linking view: instructions, data, symbol table, relocation information, and so on. Descriptions of special sections appear later in the chapter. 
> A program header table tells the system how to create a process image. Files used to build a process image (execute a program) must have a program header table; relocatable files do not need one. A section header table contains information describing the file's sections. Every section has an entry in the table; each entry gives information such as the section name, the section size, and so on. Files used during linking must have a section header table; other object files may or may not have one.

However, in the actual implementation, I didn't use the definition procvided; instead, I drew inspiration from the xv6 design

```c
// ELF Header
struct elfhdr {
  uint32 magic; // must be ELF_MAGIC
  uint8 elf[12];
  uint16 type;    // file type
  uint16 machine; // 0xF3 for RISC-V
  uint32 version;
  uint64 entry; // gives the virtual address to which the system first transfers
                // control, thus starting the process
  uint64 phoff; // holds the program header table's file offset in bytes
  uint64 shoff; // Offset from the beginning of the file to the section header
  uint32 flags; // holds processor-specific flags associated with the file
  uint16 ehsize;    // holds the ELF header's size in bytes.
  uint16 phentsize; // holds the size in bytes of one entry in the file's
                    // program header table
  uint16 phnum;     // holds the number of entries in the program header table
  uint16 shentsize; // The byte size of each entry
  uint16 shnum;     // Display the number of entries in the section table
  uint16 shstrndx;
};```


The byte alignment is the same as above, except that the magic numbers have been removed to make comparison easier.

I've isolated the important fields:

`e_phoff`: This member holds the program header table's file offset in bytes. If the file has no program header table, this member holds zero.
`e_phnum`: This member holds the number of entries in the program header table. Thus the product of e_phentsize and e_phnum gives the table's size in bytes. If a file has no program header table, e_phnum holds the value zero.

typedef struct {
	Elf64_Word	p_type;

	Elf64_Off	p_offset;
	Elf64_Addr	p_vaddr;
	Elf64_Addr	p_paddr;
	Elf64_Xword	p_filesz;
	Elf64_Xword	p_memsz;
	Elf64_Xword	p_align;
} Elf64_Phdr;
```


```c
// Program Header
struct proghdr {
  uint32 type;
  uint32 flags;
  uint64 off;    // gives the offset from the beginning of the file at which the
                 // first byte of the segment resides
  uint64 vaddr;  // gives the virtual address at which the first byte of the
                 // segment resides in memory
  uint64 paddr;  // On systems for which physical addressing is relevant, this
                 // member is reserved for the segment's physical address
  uint64 filesz; // gives the number of bytes in the file image of the segment
  uint64 memsz;  // gives the number of bytes in the memory image of the segment
  uint64 align;
};
```

`p_filesz`: The number of bytes actually occupied by this segment in the disk file.
`p_memsz`: The total number of bytes occupied by this segment after it is loaded into virtual memory.

In the vast majority of cases, `memsz` is greater than or equal to `filesz` because of the presence of the BSS segment. To save disk space, this segment is compressed, but once it is loaded into memory, it must be reallocated.

I feel like what I don't quite understand is the pagination issue.

I just realized today that PTE_U isn't mapped in the entire upper half of my kernel space.

The memory allocated using `kalloc` is first fully mapped by our kernel, so the addresses returned by `kalloc`—that is, the virtual addresses (VA)—can be accessed directly by accessing the upper half of the address space, since there is no PTE_U validation.

So this explains one thing: when paging is enabled, all addresses become virtual addresses, and the so-called PA and VA both become virtual addresses.

Therefore, when mapping the page table for user mode, we choose to map it to PA (the so-called low address). This way, when the kernel needs to access a user-mode address, it can do so directly by accessing the high half of the address space—the region not marked with the PTE_U flag.

So, you should know that once paging is enabled, any memory allocated via `kalloc` has already been mapped by the kernel in advance. Any subsequent mapping will result in duplicate entries, as the kernel always maintains one copy marked with `PTE_R` and `PTE_W`.
