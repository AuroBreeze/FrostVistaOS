{
We still encountered quite a few issues while implementing the fork.

Frist, should we emulate the page table copying functionality in xv6? In xv6, to implement pagetable copying, the approach chosen was to use PGSIZE to facilitate the process. However, this method has a drawback: what if I have a sparse layout? Relying solely on size constraints to limit the number of PGSIZE operations will not allow me to locate all of my pagetable.

So, the best approach is to simply iterate through the page table, locate the corresponding entry, and copy it directly.
However, it is important to note that since this is a complete copy of the page table, we must also collect the virtual addresses from the page table so that we can map them to their corresponding locations.

Another point to keep in mind is that the addresses we map in the page table are physical addresses—that is, low addresses. Therefore, when we want to obtain the pointer to the next page table, we must convert it to a virtual address, which is a high address.

Additionally, when mapping, we must first allocate space and copy the data there.
}

{
There are also a few things to keep in mind when implementing a fork: after the fork is complete, we must set the child process’s A0 register to 0 and handle the child process’s SPEC, otherwise the child process will end up spinning in place.
}

{
However, the biggest issue I encountered today was that the linker placed the global and static variables at low memory addresses.

When I encountered this problem, I was really puzzled. Why did adding just one element to the syscalls array render it unusable? The `write` function was working fine, but after adding `fork`, it stopped working—and I had no idea what was causing it.

This touches on linker concepts; the addressing modes for a single element and multiple elements may differ, so I might need to brush up on this.
}
