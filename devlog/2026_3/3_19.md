{
Since I moved the global and static variables to lower memory addresses yesterday, I modified the ld file so that they are resolved to higher addresses during parsing rather than being manually converted.

Virtual memory addresses (VMA) and load addresses (LMA) have been configured in LD.

This configuration allows kernel code and other components to be placed in the low-address space, while high-address space is read during parsing.

Interestingly, however, because `-mcmodel=medany` is used, addresses in the LD file are still resolved as PA (low addresses) when the OS first boots. 
}

{
Why does using `swtch` force a process switch? Doesn't it just switch the context? Does the usertrapret exit still use the original trapframe's EPC?

The key point here is the `ret` instruction in the `switch` statement. Since `switch` loads new data into the registers, `ret` will return based on the newly loaded values of `SP` and `RA`. Therefore, the return point should be where the process was last suspended—which should be the `yield` statement.

And when he returned, he used his own trapframe.

Of course, this might be a bit confusing. Keep in mind that all our programs are stored in memory as binary code, and the same goes for the SP. This switch statement directly moves the SP to the location that currently belongs to it. Also, remember that we previously allocated space specifically for our stack, so when we jump to our own stack space, pushing onto the stack will not corrupt the original stack.
}
