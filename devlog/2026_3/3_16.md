{
While dealing with the trap bug yesterday, I was still wondering why I couldn't enter the j . loop after using ecall. It kept triggering a cause 5 trap, which I misidentified as an exception, but is actually a Supervisor Timer Interrupt.

After tearing my hair out, I finally realized my fatal flaw: I hadn't separated interrupt handling from exception handling, treating everything as an exception.

My initial mental model was flawed. I assumed that usertrap was exclusively for exceptions, and that clock interrupts would gracefully "wait in line" until usertrap finished and returned to S-mode.

The harsh reality of RISC-V is different: once an interrupt (like a timer) is delegated from M-mode to S-mode, it is always globally enabled from the perspective of U-mode. It doesn't wait; it transparently preempts U-mode execution. Therefore, the very first thing a trap handler must do is check the highest bit of scause to route interrupts and exceptions separately.
}

{
Naturally, a few unexpected issues still popped up while implementing usertrap.

It's crucial to remember that the moment a trap is triggered, the hardware automatically escalates the privilege level (from U-mode to S-mode) before handing over control.

This means that the exact moment we enter the C function usertrap, we are already operating in S-mode. Because of this, our very first job is to overwrite stvec to point to kernelvec. If we don't, and a timer interrupt or page fault occurs while we are inside the trap handler, the CPU would jump right back to the beginning of the user trap vector, completely destroying the kernel context.

After securing the vector, we can safely acknowledge the clock interrupt and schedule the next timer tick.
}


{
Returning to user space via usertrapret requires equally careful choreography.

Since we just repointed stvec to the kernel vector, we need to point it back to uservec before returning to U-mode. However, this must be done with interrupts strictly disabled (intr_off()). If a timer ticked exactly after we wrote uservec but before we actually returned to U-mode, the kernel would falsely drop into the user trap handler. That would be quite a spectacular crash.

Next, we need to advance the program counter (sepc). While doing this, we must be mindful of instruction lengths (though ecall is always 4 bytes, other traps might involve 2-byte compressed instructions).

Finally, we configure the status bits. We clear the SPP bit (to return to U-mode) and set the SPIE bit. We don't execute sret directly in C. Instead, we let the final assembly routine handle it. When sret is finally executed, the hardware will copy the SPIE bit into the SIE bit, automatically re-enabling interrupts exactly as the CPU lands back in user space. For the deep magic behind SPIE, the RISC-V Privileged Architecture Manual is your best friend.
}
