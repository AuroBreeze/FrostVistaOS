# 操作系统中进程间锁转移的相关问题

## Forward

在进程执行的过程中，锁的切换是非常频繁的。

在这个过程中，锁的所有权会从一个进程转移到另一个进程。
因此，我想针对这个细节做一些说明。

## Lock

锁的作用是什么？

简单来说，是为了保护数据，防止其受到进程间竞争带来的干扰。原理听起来很简单，但实际实现起来却并不那么容易。

## Start

首先是从`user_init()`函数开始，它负责加载进程。当然，在这个阶段它并不需要持有锁。
接着，`scheduler()`会准备执行该进程。

它会持有进程的锁，遍历进程表，并定位到需要执行的进程。如果该进程尚未准备就绪，它就会释放锁并等待，直到进程变为就绪状态。

假设进程已经就绪，它将进入`swtch`函数，此时会跳转到`usertrapret`。如果在`usertrapret`中持有了锁，它将会被释放。完成这些操作后，进程就会开始正常运行。

它会持有进程的锁，遍历进程表并找到需要执行的进程。如果进程未就绪，它会释放锁并等待，直到自身变为就绪状态。

假设进程已经就绪，它将进入`swtch`，此刻它会跳转到`usertrapret`，如果在`usertrapret`中持有了锁，则将其释放，完成这些之后，进程就会开始正常运行。

> 不过需要注意的是，这个时候，在CPU运行的进程中，当前进程的锁在最后还是需要被释放的，但是这个时候，锁已经释放了，所以后续这个程序停止的时候，必须再次申请锁才可以。

在正常运行的过程中，假设遇到了`Tick`中断，那么就会陷入到`usertrap`中，因为处理完`Tick`后会使用`yield`，所以在`yield`中会先申请锁，进入到`sched`中，`sched`会检查锁，如果在没有持有锁的情况下进入，就会触发`panic`，因为并不是只有`yield`中会使用`sched`，所以这里需要考虑持有锁的状态。

这个时候又会遇到`swtch`，同样的，yield中的`release lock`又没有了匹配的的锁来进行释放，`swtch`会切换到上一次CPU没有运行完的地方，也就是在`scheduler`中进行的`swtch`的操作，然后跳过这个进程，并释放这个进程的锁，而这个进程的锁，是我们在上一段`yield`中申请的，然后CPU继续遍历数组，寻找到下一个进程，并开始运行。

我们这里简单起见，认为只有内核进程和一个用户进程。

我们找到的下一个进程依然还是我们上面的进程，先申请锁，这个进程会继续执行(而且我们上面所说的，这个进程还欠了一个`release lock`的操作)，寻找到这个进程后，通过`swtch`跳转到上一次没有运行完的地方，也就是`yield`中，这个时候，`yield`会释放，我们在进行寻找中申请的锁，然后处理完`Tick`中断，从`usertrap`到`usertrapret`这里因为没有锁并没有什么问题，所以就跳过了。

我们假设，这个进程就这样运行完，调用了`sys_exit`函数调用，这个系统调用会进入中断，直到`exit`执行，在`exit`中会先申请当前进程的锁，然后通过`sced`切换到新的进程，也就是通过`swtch`切换到CPU的执行处，这个循环结束，调用`release lock`释放掉`exit`中申请的锁。

这就是一个锁转移过程。


```
Scheduler Context (CPU)         Lock State            Process Context (User/Kernel)
===========================   ====================   ===================================
user_init()加载进程
          |
scheduler() 准备执行
          |
  [Acquire] 获取进程锁 -------> [ 锁定 (LOCKED) ]
          |
  找到就绪进程
  swtch() 切换上下文 -------------------------------------> (首次运行跳转)
                                                                 |
                                                           usertrapret()
                                [ 解锁 (UNLOCKED) ] <------ [Release] 释放进程锁
                                                                 |
                                                           正常运行 (User Space)
                                                                 |
                                                           [Tick 中断发生]
                                                           usertrap()
                                                                 |
                                                           yield()
                                [ 锁定 (LOCKED) ] <-------- [Acquire] 获取进程锁
                                                                 |
                                                           sched() (检查锁状态)
                                                                 |
                                                           swtch() 切换上下文
(返回到上一次切换点) <----------------------------------------------|
          |
  [Release] 释放进程锁 -------> [ 解锁 (UNLOCKED) ]
          |
  (遍历进程表，寻找下一个)
  
==================== 假设寻找到的下一个进程依然是它 ====================

scheduler()
          |
  [Acquire] 获取进程锁 -------> [ 锁定 (LOCKED) ]
          |
  swtch() 切换上下文 -------------------------------------> (从上次 yield 处恢复)
                                                                 |
                                                           yield() 结束
                                [ 解锁 (UNLOCKED) ] <------ [Release] 释放进程锁
                                                                 |
                                                           usertrapret()
                                                                 |
                                                           正常运行 (User Space)
                                                                 |
                                                           [调用 sys_exit]
                                                           exit()
                                [ 锁定 (LOCKED) ] <-------- [Acquire] 获取进程锁
                                                                 |
                                                           sched() 
                                                                 |
                                                           swtch() 切换上下文
(返回到 scheduler) <------------------------------------------------|
          |
  [Release] 释放进程锁 -------> [ 解锁 (UNLOCKED) ]
          |
    (进程终止流程)
```



## Further Reading

由此延伸到`wait`函数中，在`wait`中通常会使用睡眠锁来减少CPU的忙等，所以在处理`wait`的时候，是需要先获取当前进程的锁的，在没有等待到对应对子进程结束的时候，`sleep`会调用`sched`来切换到新的进程中，而`sched`又会检查锁的状态，所以必须申请锁。

> 为什么`sched`必须要报纸持有锁？
> 因为`sched`切换后会进入到`schduler`中，如果没有锁的情况下结束当前的遍历到的进程，会导致`release`无锁可放，触发`panic`
> `sched`的主要作用就是切换新的进程
