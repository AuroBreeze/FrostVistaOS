I used to find the use of pipes confusing when working on xv6 exercises, but today I’ve finally gained a thorough understanding of how they work.

The issue I encountered earlier was why the unused end of a pipe needs to be closed when using it in a child process. At the time, I was told that if the read end is being used, the write end should be closed first because an EOF signal can only be received after the write end is closed; however, it turns out that isn't the actual reason.

We know that when a child process is created, `fork` copies the data to the child, and the child also inherits the file descriptors (FDs) for the pipe structure; consequently, there are two sets of references to the pipe. While closing the read or write end in the child process primarily serves to promptly close unnecessary references (such as the input end in `echo 123 > cat`), leaving an extra write-end reference open—whether in the child process or the parent process—can lead to serious consequences: specifically, the `cat` command waits for an EOF signal to terminate, but if it fails to close its own write-end reference, it will never receive that EOF and will hang indefinitely.

I also entertained a mistaken idea: since the goal was to close redundant or useless file descriptors, couldn't I simply wait for `exit` to close them—provided the `exit` logic was implemented correctly? However, this approach is problematic, as mentioned at the end of the previous section.

If the write end on the right side is not closed, the process will still hang—even if the left side and the parent process have closed their write ends—because the right side is waiting to receive an EOF signal indicating the write end has closed, yet it has not closed its own write end and is effectively waiting for itself to exit and close it.

So, there is no magic to pipes; simply understanding file references, the closing mechanism, and the pipe's own closing mechanism allows you to gain insight into the pipe's kernel-level implementation.
}
