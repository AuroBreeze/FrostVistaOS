{
I've been working on a file system lately, and I'm still struggling with reading data.

The problem is: Currently, to have the OS read data from the file system, we have to rely on the process ID and use spin locks and sleep locks, but I need to read the data first before running the first user process.

My original plan was to directly initialize a fake first process, but I discovered a bug. Specifically, after allocating the process using `alloc_process`, I modified `context->ra` to point to the kernel function I had initialized in my filesystem, and then used `exit` to terminate the process. However, this method still has issues. This method is highly dependent on the program’s execution speed. When execution is fast, no interrupt is triggered, allowing the initialization to complete quickly and move on to the next task. However, if execution slows down even slightly, a clock interrupt will switch control to the user process. Since this initialization process did not use `usertrapret` to return to user mode, this causes the user process to encounter an unexpected interrupt that does not originate from user mode.

So we need to find a new way to address this issue.

I noticed that the solution for xv6 involves setting `ra` to `forkret` when using a process allocated via `allocprocess`—which effectively implements `read` and `usertrapret`. However, I also noticed that this function uses a built-in static integer for the check, which doesn’t seem very elegant.

I'm thinking about whether to try writing a kernel process in the kernel specifically for initializing and recycling user processes. It might be worth a try.

I always thought this should work, but unfortunately, it doesn't.

{
The current solution to the problem where the first process uses `exec` but relies on `bread` and locks is the method used earlier to load processes: compiling the `.S` file into an array and loading it. However, this time, we call only one `exec` system call and use `exec` for initialization.

But for some reason, I always get confused about the difference between loading and executing code. I even hesitated when switching back to using `exec` with an array to load the code, worrying that `exec` might cause the system to freeze due to a buffer overflow. But in reality, the array simply loads the `exec` call; the call isn’t executed until it enters the scheduler. By that point, the process has already started, the environment is already set up, and with `ra` configured, the filesystem can be initialized during the first load. My thinking here, just like above, was confused between loading and execution. You have to understand that when the scheduler dispatches for the first time, the program will inevitably enter the location pointed to by `context.ra`, and at that point, the environment is already in place.
}
